## 分项评审

### 1. 方案完整性
当前方案已覆盖：
- 域名切换
- WebView 页面重构
- 模板残留清理
- 测试和文档修复

仍需补充：
- 明确新增测试缝隙，避免在 Widget Test 中直接实例化真实 `WebViewController`
- 明确 `onNavigationRequest` 或同类导航约束，避免跳转到非目标主机或异常 scheme
- 明确错误页/重试机制，避免页面加载失败时只有日志没有可见反馈
- 明确根目录 HTML 文件处理策略，只修正文案型旧域名，不把它们误当成运行时代码重构

### 2. 技术选型合理性
- 继续使用 `webview_flutter` 是合理的，当前问题不是技术栈不足，而是工程组织失衡。
- 通过 Dart 常量、配置类、辅助函数拆分脚本和路由，也符合“小步重构、避免过度设计”的目标。
- 不建议本轮引入状态管理库、路由库或复杂 DI，收益不足。

### 3. 实施步骤可执行性
- 步骤整体可执行，但建议把“测试改造”前置到核心重构设计中，而不是最后补救。
- 推荐实施顺序调整为：
  1. 抽离配置和导航目标
  2. 抽离注入脚本与页面策略
  3. 重写 WebView 页面状态处理
  4. 修复测试与文档
  5. 更新平台元数据和剩余旧域名

### 4. 潜在风险与更优方案
- 风险 1：站点 DOM 变动后，现有大量选择器可能失效。
  - 建议：把注入脚本分段，并对关键页面类型做显式判断，减少“一次性注入全部规则”的副作用。
- 风险 2：当前测试无法运行。
  - 建议：把 URL 构建、导航配置、脚本拼装提炼为纯 Dart 单元可测逻辑；Widget Test 只测应用壳，不直接依赖真实 WebView 平台实现。
- 风险 3：`onPopInvoked` 已弃用。
  - 建议：直接迁移到新 API，避免刚重构完又留下 analyzer 告警。
- 风险 4：桌面平台虽然存在模板文件，但当前主要构建脚本是 Android APK。
  - 建议：桌面平台只做低风险品牌信息同步，不做包名或工程结构级重命名。

### 5. 明确修改建议
- 建议 1：Plan v2 中新增“测试策略设计”小节，写明哪些逻辑做单元测试、哪些保留手测。
- 建议 2：Plan v2 中新增“导航约束与错误恢复”内容，覆盖异常链接、加载失败和重试入口。
- 建议 3：Plan v2 中明确品牌名称，例如统一使用 `2026Copy`，避免 `copy_fullter`、`Copy Fullter`、`2025Copy` 混用。
- 建议 4：Plan v2 中明确根目录 HTML 文件的处理边界，只做旧域名文本修正或保留说明，不做结构重写。
